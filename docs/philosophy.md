# Mold Philosophy & Principles

> 이 문서는 Mold 프로젝트의 존재 이유, 비전, 그리고 오랫동안 거의 변하지 않을 핵심 철학과 원칙을 담고 있습니다. 구현 방식이나 파일 구조 같은 가변적인 세부사항은 포함하지 않습니다.

---

## 1. 존재 이유 (Why Mold Exists)

대부분의 온라인 서비스는 **사용자 입출력, 데이터 저장, 권한 제어, 폼 UI 렌더링**이라는 공통된 기본 구조를 공유합니다. 하지만 수많은 프로젝트가 이 똑같은 구조를 재작성하고, 설정 파일을 정의하며, 레이어 간의 파편화된 매핑 코드를 작성하는 데 시간을 허비합니다.

Mold는 **Resource 정의 하나로 온라인 서비스의 기본 구조를 완벽하게 자동 실행하는 Runtime**입니다. Mold는 모든 형태의 복잡한 분산 아키텍처를 지원하려 하지 않습니다. 대신 가장 흔하게 발생하는 온라인 서비스의 반복 구조를 결정론적으로 자동화하여, 개발자가 본질적인 비즈니스와 사용자 경험에만 집중하도록 돕습니다.

---

## 2. 핵심 비전 (Vision)

1. **AI와 사람이 함께 만들기 쉬운 환경**
   * 선언적 명세(Resource YAML)를 통해 사람과 AI 에이전트가 동일한 단일 소스를 공유하며, 코드 수정 없이 서비스를 확장할 수 있게 합니다.
2. **반복적인 백엔드 작업의 결정론화**
   * 동일한 명세는 어떤 환경에서든 항상 동일한 백엔드 구조를 만듭니다.
3. **도메인과 비즈니스에의 집중**
   * 개발자를 반복적인 CRUD 작업에서 완전히 해방시켜 프론트엔드 UX와 비즈니스 가치 창출에 집중하게 합니다.
4. **쉬울 뿐만 아니라 "실패하기 어려운" 서비스 (Hard to Fail)**
   * 빠른 서비스를 만드는 것을 넘어, 부팅/리로드 시점의 원자적 검증(Fail-Fast Compile), 하강형 권한 가드, 안전한 삭제 정책을 통해 사용자가 실수하더라도 서비스가 쉽게 고장 나지 않도록 보호합니다.

---

## 3. 핵심 철학 및 원칙 (Core Principles)

### ① Resource가 유일한 단일 진실 공급원(Source of Truth)이다.
모든 데이터 스키마, API, 유효성 검사, 권한, 기본 View는 오직 Resource 정의로부터 파생됩니다. Resource 외의 다른 정보원에 의존하지 않습니다.

### ② 결정 가능한 것은 모두 결정한다 (설정보다 생성).
가능한 한 옵션이나 설정(Configuration)을 만들지 않습니다. Resource 정의만으로 추론할 수 있는 것은 런타임이 결정적으로 추론하여 자동 생성합니다. 설정은 Escape Hatch로만 존재합니다.

### ③ Backend는 창의적인 공간이 아니다 (결정적 백엔드).
동일한 Resource 정의는 항상 동일한 결과를 만들어내야 합니다. 백엔드는 개발자의 자유도를 발휘하는 공간이 아니라 명세를 이행하는 "컴파일러"로 동작합니다.

### ④ IR은 Target에 독립적이어야 한다.
Resource가 컴파일된 중간 표현(IR)은 실행 타깃(Go, Cloudflare Workers 등)의 세부사항이나 언어 종속적 힌트를 포함하지 않는 순수한 도메인/보안 명세이어야 합니다.

### ⑤ Target은 가능한 한 단순하고 멍청해야 한다 (Dumb Target - Smart Spec).
실행 타깃(Target)은 비즈니스 이유를 스스로 해석하지 않습니다. 오직 주어진 명세를 자신의 플랫폼 어휘로 단순 변환 및 서빙만 수행합니다.

### ⑥ 개발자는 Generate를 의식하지 않아야 한다 (Invisible Infrastructure).
Mold는 백그라운드에서 소스를 관찰하고 컴파일하는 "투명한 컴파일러"로 동작해야 합니다. 개발자는 복잡한 CLI 생성 명령을 외울 필요 없이, 소스 파일 저장만으로 결과를 즉시 확인합니다.

### ⑦ 기존 코드를 함부로 추상화하지 않는다 (마세라티 원칙).
아직 발생하지 않은 미래의 문제를 미리 해결하지 않습니다. 
우선순위: **1. 단순함 ➔ 2. 빠른 개발 ➔ 3. AI 친화성 ➔ 4. 개발 생산성 ➔ 5. 확장성**

### ⑧ 반복되는 패턴이 실증되기 전에는 새로운 계층을 만들지 않는다.
추상화는 머릿속에서 예쁘게 설계하는 것이 아니라, 실제 서비스를 제작해 보며 반복되는 문제 패턴을 현장에서 "발굴(Discover)"하는 것입니다.

---

## 4. 하지 않는 것 (Non-Goals)

Mold는 다음을 목표로 하지 않습니다.

* 거대한 Enterprise Framework
* 분산 시스템 및 Microservice Platform
* Workflow Engine
* Frontend Framework
* 범용 ORM
